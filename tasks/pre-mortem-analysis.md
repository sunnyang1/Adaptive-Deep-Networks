# Pre-mortem 分析: TurboQuant 重构

## 场景设定
重构项目失败，导致以下问题：

## 失败原因分析

### 1. 循环导入修复导致模块依赖混乱
**原因**: 简单移动导入到顶部破坏了原有的延迟加载机制，导致启动时循环依赖错误
**预防**: 
- 使用依赖注入模式替代直接导入
- 添加模块依赖图分析
- 逐步重构，每次验证

### 2. QJL 性能优化引入数值精度问题
**原因**: 矩阵运算优化改变了计算顺序，累积误差导致 attention 权重不准确
**预防**:
- 添加数值稳定性测试
- 与原始实现对比输出
- 使用更高精度中间计算

### 3. 大规模重命名导致 API 破坏
**原因**: `PolarQueryAdapter` 重命名影响了外部依赖
**预防**:
- 使用兼容层/别名保持向后兼容
- 搜索所有引用点
- 添加弃用警告

### 4. 类型注解修改破坏动态类型使用
**原因**: 某些调用点依赖动态类型特性
**预防**:
- 渐进式添加类型注解
- 使用 `Any` 作为 fallback
- 完整测试覆盖

### 5. 测试覆盖率目标不切实际
**原因**: 某些死代码或复杂边界条件难以测试
**预防**:
- 合理设置覆盖率目标 (75%)
- 优先测试核心路径
- 标记不可测试代码

## 关键决策点
1. **是否保持向后兼容**: 是，使用别名
2. **是否一次性重构**: 否，分阶段
3. **是否添加新依赖**: 否，仅工具类
